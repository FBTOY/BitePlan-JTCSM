## Project overview

家庭厨房助手 is a Next.js web app that turns a user's kitchen context (available ingredients, tools, skill level, time budget, dietary notes) into a detailed, step-by-step cooking plan generated by an AI model. Users can follow the plan as an interactive checklist, track progress, submit feedback at the end, manage a pantry, and browse cooking history.

## Common commands

- `npm run dev` — Start the development server on http://localhost:3000.
- `npm run build` — Build the app for production.
- `npm run start` — Start the production server (run `build` first).
- `npm run lint` — Run ESLint.
- `npm test` — Run the full Jest test suite once.
- `npm run test:watch` — Run Jest in watch mode.
- `npx jest __tests__/recipe.test.ts` — Run a single test file.

## AI provider configuration

The app does not rely on server-side environment variables for API keys. Instead, users configure AI providers in the UI at `/settings` and the credentials are stored in `localStorage`.

Supported providers:

- **Moonshot (Kimi)** — OpenAI-compatible API. Default base URL: `https://api.moonshot.cn/v1`. Default model: `moonshot-v1-8k`.
- **Anthropic (Claude)** — Default model: `claude-3-5-sonnet-20241022`.

When generating a recipe, `app/page.tsx` sends the active provider config to `app/api/recipe/route.ts`, which calls the appropriate SDK.

## High-level architecture

### Frontend (App Router)

- `app/page.tsx` is the single-page client shell. It owns the global view state (`profile` → `dish` → `recipe` → `cooking` → `feedback` → `done`) and localStorage persistence through `lib/storage.ts`. It avoids `setState` inside effects by initializing state from `localStorage` in `useState` and using `lib/hooks.ts` for hydration-safe mount detection.
- `app/settings/page.tsx` — configure AI providers (Kimi/Claude API keys and models).
- `app/pantry/page.tsx` — manage pantry items (ingredients, seasonings, tools).
- `app/history/page.tsx` — browse archived cooking sessions and feedback.
- `components/` contains the major UI sections:
  - `KitchenProfileForm.tsx` — servings, time, skill level, dietary notes, and a link to pantry management.
  - `DishInput.tsx` — asks what the user wants to cook.
  - `RecipeCard.tsx` — displays the generated recipe, ingredient/tool gaps, and lets users add ingredients/seasonings to the pantry.
  - `CookingMode.tsx` — interactive step-by-step checklist with progress tracking.
  - `FeedbackForm.tsx` — rating, difficulty, taste, and notes.
  - `ProviderConfigForm.tsx` — add, edit, and select AI providers.
  - `PantryManager.tsx` — add/remove pantry items.

### Backend

- `app/api/recipe/route.ts` is the only API route. It receives the kitchen profile, active provider config, and dish name, then delegates to `lib/recipe.ts` to call the selected AI and return a validated `RecipePlan`.
- `lib/recipe.ts` builds the prompt, validates the JSON response with Zod, and switches between the Anthropic SDK and the OpenAI-compatible SDK for Moonshot/Kimi.

### Data model and persistence

- `lib/types.ts` defines `KitchenProfile`, `RecipePlan`, `RecipeStep`, `CookingSession`, `SessionFeedback`, `ProviderConfig`, and `PantryItem`.
- `lib/storage.ts` reads and writes profile, current session, archived sessions, provider configs, and pantry items from `localStorage`.
- `pantryToProfile(profile, pantry)` merges pantry items into the kitchen profile before recipe generation.
- Sessions are archived when cooking is completed.

### Tests

- Jest is configured in `jest.config.ts` using `next/jest` with `jsdom`.
- Tests live in `__tests__/` and cover recipe prompt building/JSON parsing, localStorage helpers (including providers and pantry), and the `DishInput` and `PantryManager` components.

## Notes

- This project was bootstrapped with `create-next-app` and uses Next.js 16, React 19, TypeScript, Tailwind CSS 4, and the App Router.
- The app is designed to work without a database in the MVP; all user data is stored in `localStorage`.
